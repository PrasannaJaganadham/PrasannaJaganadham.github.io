● Shipped · Apr 2026 02 Python · web scraping

Amazon price tracker & email alerter

A Python project I built end-to-end to learn web scraping properly. Tracks a product on amazon.fr daily, logs to CSV, and emails me when the price drops below my target.

daily
auto price check
Expand ▾
Problem
I wanted to learn web scraping by solving a real problem. I was watching a Pokémon Charizard T-shirt on amazon.fr waiting for it to drop below €20 — so I built a script to check the price daily and email me when it does.
Data
Live Amazon.fr product page HTML · scraped daily · title + price extracted to CSV with timestamp. Small dataset — but it's mine, not a Kaggle download.
Approach
  • Scraping: requests with custom browser headers (User-Agent, Accept-Language) to avoid bot detection
  • Parsing: BeautifulSoup to extract title and price using a-price-whole + a-price-fraction selectors (the old priceblock_ourprice ID no longer exists — debugged via DevTools)
  • Locale handling: amazon.fr shows prices as "22,00" — wrote cleanup for the French comma decimal
  • Persistence: csv module, append-only, header written once
  • Alerting: smtplib + Gmail SMTP with an App Password; triggers when price < €20
  • Scheduling: while True loop with 24-hour sleep
What I learned
Tutorials go out of date fast. Amazon's HTML had changed since the tutorial I followed — the price selector was broken. I had to inspect the real page in DevTools to find new classes. That debugging skill (read the real HTML, don't trust the tutorial) transfers to every scraping job.
Honest scope
This is a learning project, not production. Tracks one product. Runs on my laptop. But every part — headers, parsing, CSV, email, scheduling — works end-to-end. Extending next to track multiple products across amazon.fr / .de / .co.uk.
Tech stack
Python 3.13 BeautifulSoup4 requests pandas smtplib (SMTP) csv Jupyter
</> GitHub repo + README 📓 Jupyter notebook
The README documents the 6 real bugs I hit (Amazon blocking, broken selectors, French comma decimals, Gmail App Passwords, CSV duplicate headers, infinite loop stuck in Jupyter) and how I fixed each one. That debugging trail is the project.
◆ Academic · MSc project 03 Power BI · DAX

Spotify music analytics dashboard

A Power BI dashboard analysing music trends, artist performance, and listener engagement using dynamic slicers, custom DAX measures, and KPI cards. Built during the MSc to practise data modelling end-to-end.

DAX
interactive dashboard
Expand ▾
Problem
Music streaming generates huge amounts of engagement data, but most of it sits in static tables. The goal: turn a Spotify dataset into an interactive dashboard where anyone could slice by genre, era, or artist and get answers in seconds.
Data
Public Spotify tracks dataset (Kaggle) — hundreds of tracks with audio features (energy, danceability, tempo, acousticness, etc.), stream counts, artist metadata, and release year.
Approach
  • Data modelling: star schema with tracks fact table + artist / genre / era dimensions
  • DAX measures: custom measures for stream share, average energy by era, YoY artist growth, top-N by genre
  • Interactivity: slicers for genre, release year, artist; drill-through from artist → tracks
  • Visuals: KPI cards, radial audio-feature profiles, time-series, ranked bar charts
Key insight
Audio features cluster strongly by era — energy and loudness have trended upward over decades, while acousticness drops sharply after 2000. Useful for anyone building era-based recommendation logic.
What it taught me
Power BI fundamentals end-to-end — star schema design, DAX measure patterns, and interactive slicer design. This was my deepest hands-on Power BI work, building on what I started during the YBI internship.
Tech stack
Power BI DAX Data modelling (star schema) Power Query

◆ Academic · MSc project 04 SQL · Docker · statistics

COVID-19 ICU & Med/Surg patient load analysis

A dockerised SQL database monitoring ICU and surgical patient loads, with hypothesis testing to identify over-capacity risk patterns across hospital facilities.

p < 0.05
risk patterns identified
Expand ▾
Problem
Public health teams needed a way to compare ICU and Med/Surg capacity utilisation across facilities — and statistically identify which hospitals were running persistently over capacity.
Data
Public US hospital capacity data — daily bed utilisation across hundreds of facilities over the pandemic window.
Approach
  • Built a Dockerised PostgreSQL schema for reproducibility
  • SQL aggregations for occupancy ratios by facility and week
  • Hypothesis testing (two-sample t-tests with multiple-comparison correction) on facilities exceeding capacity thresholds
  • Output: ranked list of facilities + confidence intervals for operational follow-up
Key insight
Over-capacity was not random — a subset of facilities showed statistically significant, persistent strain rather than random spikes. That changes the operational response from "add beds" to "rebalance intake."
What I'd do differently
Academic project — the next step would be tying the stats output to a dashboard the hospital ops team could use weekly. Closing that loop is what separates analysis from impact.
Tech stack
PostgreSQL Docker Python (scipy.stats) Jupyter
